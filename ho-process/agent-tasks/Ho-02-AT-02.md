# Ho-02-AT-02 — Ordered chains: the client layer

Parent ho: `ho-process/ho-02-ordered-chains.md`. **Depends on `Ho-02-AT-01`** —
there is no chain strip to key against or style until the floor renders it. Run
AT-01 first.

## Goal

Add the two client-side affordances on top of the rendered chain strip: keyboard
prev/next, and styling. The strip already works by mouse and by no-JS
navigation after AT-01; this task is enhancement only.

## Read first

`src/star_actually/assets/app.js` (the keyboard map in the `keydown` handler and
the `dial()` helper — chain nav is the same shape), `assets/style.css` (the
`.depth-dial` / `.depth-step` rules — the strip styles as a sibling of them).
The chain strip markup and its `chain-prev` / `chain-next` classes come from
AT-01's `fragment.html.j2`.

## Required changes

**1. `app.js` — bind `[` and `]`.**

- In the `keydown` switch, add: `[` → click the first `.chain-prev` link in
  `#content`; `]` → click the first `.chain-next` link. Model it on `dial()` /
  `goOnward()` (query within `content()`, click if present, ignore if absent or
  disabled). Respect the existing guards — `inTextField`, the modifier-key
  bail-out, the focused-anchor Enter exception.
- Per the ho's Decision 6, the keys act on the **first** chain a node belongs to
  (the first strip). Multi-chain keyboard disambiguation is deferred; the other
  strips remain mouse-navigable.
- Watch the file-length ceiling — ho-01 recorded the practitioner's ~450-line
  ruling for `app.js`, and the file is already at 469 lines, slightly over it.
  Keep the addition minimal (a `dial()`-shaped helper plus two switch cases);
  report the resulting line count in the task notes. Do not refactor the rest
  of the file to compensate — that is a separate decision for the practitioner.

**2. `style.css` — style the strip.**

- Add `.chain-strip` rules: a compact line under the depth dial, in the terminal
  palette (dim label, accent for the position, the same `--rule`/`--dim`
  treatment the dial uses). Style `chain-prev` / `chain-next` like `.depth-step`,
  with an `is-disabled` state matching the dial's disabled controls. Stacked
  strips read cleanly when a node is in more than one chain.

## Acceptance criteria

- On a built instance whose `site.yaml` declares a chain: the strip is styled and
  legible under the depth dial; `[` moves to the previous member, `]` to the
  next; at an end the corresponding key/control is inert.
- No-JS floor unchanged — with JavaScript disabled the strip's links still
  navigate (that behavior is AT-01's; confirm this task didn't regress it).
- Engine `make verify` still green (no Python touched, but run it — the build
  copies `assets/`).

## Verification

`make verify` for the Python floor, then eyeball in a browser against a built
instance that declares a chain (build it with `star-actually build`, serve
`dist/`, load a chain-member page): confirm the strip renders, `[`/`]` navigate,
and the ends disable. JS/CSS have no unit harness in this engine — the browser
check is the verification, as with the rail.
