# Ho-02-AT-01 — Ordered chains: the Python floor

Parent ho: `ho-process/ho-02-ordered-chains.md`. Read its **Think** section
first — the decisions below are settled there, not open for reopening in the
task.

## Goal

Give the engine a generic ordered-sequence primitive — a **chain** — declared in
`site.yaml` and rendered as a strip on each member node's page, with real
prev/next links. This task is the no-JS floor: everything works and is verifiable
without a browser. The keyboard and styling land in `Ho-02-AT-02`.

## Read first

`src/star_actually/config.py`, `graph.py`, `render.py`, `nodes.py`,
`templates/fragment.html.j2`. Note the two-mode contract already used for
`root_node` in `render_site` (strict raises `ConfigError`; `allow_dangling`
appends a warning) — chains follow the same contract.

## Required changes

**1. `config.py` — declare and shape-validate chains.**

- Add a frozen dataclass:
  ```python
  @dataclass(frozen=True)
  class Chain:
      id: str
      title: str
      nodes: tuple[str, ...]
  ```
- `SiteConfig` gains `chains: tuple[Chain, ...] = ()` (optional, like
  `receptionist` — backward-compatible; a `site.yaml` without `chains` stays
  valid).
- Add `"chains"` to the `known` key set. Parse `data.get("chains", [])`:
  - must be a list; each item a mapping with keys exactly `{id, title, nodes}`;
  - `id` kebab-case (reuse the same pattern `nodes.py` uses for node ids),
    unique across chains;
  - `title` a non-empty string;
  - `nodes` a non-empty list of strings with **no duplicate within one chain**.
  - Any violation raises `ConfigError` naming the chain and the fix.
- **Do not** check member existence here — `config.py` has no node set. That is
  step 3.

**2. `chains.py` — new module: membership and existence validation.**

- `@dataclass(frozen=True) class ChainLink` with: `chain_id: str`, `title: str`,
  `index: int` (1-based position, for display), `total: int`,
  `prev: str | None`, `next: str | None`.
- `links_for(node_id: str, chains: tuple[Chain, ...]) -> tuple[ChainLink, ...]`
  — every chain the node belongs to, each with its 1-based position and the ids
  of the adjacent members (`None` at an end). Order the result by the chains'
  declaration order in `site.yaml`.
- `validate_chains(chains, node_ids: set[str], *, allow_dangling: bool = False)
  -> tuple[tuple[Chain, ...], tuple[str, ...]]` — returns `(validated_chains,
  warnings)`. A member id not in `node_ids` raises `ConfigError` in strict
  mode; under `allow_dangling` it is dropped from the returned chains and a
  warning string is emitted. Mirror the `root_node` handling in `render_site`.
  The returned chains are what the renderer consumes — a dropped member must
  not survive into strip positions or prev/next targets. A chain whose members
  all drop is itself dropped (with its warning); a chain reduced to one member
  still renders (1/1, both ends disabled).

**3. `render.py` — wire it in.**

- In `render_site`, right after the graph is built (the fragments rendered in
  the node loop need the result), call `validate_chains(site.chains,
  set(graph.nodes), allow_dangling=allow_dangling)`; fold its warnings into the
  returned `RenderResult.warnings`.
- Pass the **validated** chains (not `site.chains`) down into
  `_render_fragment` as a parameter, and add
  `chains=links_for(node.id, validated_chains)` to the template context.

**4. `templates/fragment.html.j2` — render the strip.**

- Inside `.node-head`, directly **after** the `.depth-dial` div, render one
  `.chain-strip` per `ChainLink` (stack them if several). Each strip shows the
  chain `title` and `{{ link.index }}/{{ link.total }}`, plus prev and next
  controls in fixed slots:
  - available direction → an `<a>` to `{{ site.base_path }}/n/{{ link.prev }}/`
    (resp. `next`), carrying the same `hx-get=".../d2.html"`,
    `hx-target="#content"`, `hx-swap="outerHTML"`,
    `hx-push-url=".../n/<id>/"` attributes the `.nav-node` links use; give them
    classes `chain-prev` / `chain-next` and a `data-chain-id`.
  - end of chain → a disabled `<span>` in the same slot (mirror the depth dial's
    `is-disabled` pattern), so click targets never move.

## Acceptance criteria

- `site.yaml` with a `chains:` list parses into `SiteConfig.chains`; a config
  without `chains` still loads.
- Every malformed chain shape (not a list, missing/extra key, non-kebab or
  duplicate `id`, empty `nodes`, duplicate member within a chain) raises
  `ConfigError` naming the problem.
- A member id absent from `nodes/` raises `ConfigError` in strict build and
  produces a warning (not a raise) under `allow_dangling`; the dropped member
  appears in no rendered strip — positions, totals, and prev/next links are
  computed from the surviving members only.
- A built fragment for a chain member contains a `.chain-strip` with the correct
  `k/n`, a working prev/next link to the adjacent member, and disabled controls
  at the ends. A node in two chains renders two strips.
- Tests are **synthetic** — build throwaway chains/graphs in `tmp_path`, assert
  on emitted HTML. No coupling to any instance's real content.
- Engine `make verify` green: `ruff format --check`, `ruff check`, `mypy
  --strict`, `pytest` at ≥90% coverage.
